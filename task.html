01 · Context

The Problem

Understand this before you prompt your first line of code.

Company X is actively hiring across multiple roles. Their entire recruitment process runs on spreadsheets and paper notes — hiring managers update a shared Excel file by hand, and there's no single source of truth.

The result: candidates fall through the cracks. Someone forgets to update a stage, an email never gets sent, or a promising CV gets buried in an inbox. By the time anyone notices, the candidate has moved on.

They need a simple web application where the hiring team can see every open role, drill into each one, and manage candidates through a visual pipeline — dragging cards from stage to stage as the process moves forward. Every action should be timestamped so nothing gets lost or forgotten.

02 · Screens

What to Build

Four screens. All four must be functional, connected to the real API, and deployed live.

SCREEN 01

Jobs Dashboard

Lists all open positions. Each job card shows its title, department, and active candidate count. Clicking a job opens its pipeline.

SCREEN 02

Candidate Pipeline

A Kanban board scoped to one job. Five columns: Applied → Shortlisted → Interviewing → In Assessment → Accepted. Cards show photo + name and can be dragged between columns. Every move is saved and timestamped.

SCREEN 03

Candidate Detail

Clicking a card opens the full application — name, photo, role applied for, date applied, current stage, and a chronological history of every stage change.

SCREEN 04

Rejected Archive

A read-only page listing every rejected candidate: name, role, the stage they were rejected at, and exact date and time. Once rejected, a candidate cannot be dragged back into the pipeline.

03 · Requirements

What's Expected

Core requirements must be met. Bonus items are genuinely optional — a solid core beats a buggy everything.

AI & Documentation
  • PROMPTS.md — Document your key AI prompts: initial scaffolding, data model decisions, drag-and-drop implementation, and any debugging sessions. Show the evolution, not just final prompts.
  • AI_REVIEW.md — Critique your own codebase: 2 things the AI did well, 2 things you had to manually fix or rewrite, and 1 technical debt item you'd address next.
  • Decision audit — In your README, explicitly note which architectural decisions were AI-suggested vs. your overrides. We want to see your judgment, not just AI output.
Backend
  • REST API in Node.js / Express with consistent JSON responses throughout.
  • Data model covering Jobs, Candidates, Applications (the link between a candidate and a job), and a StageHistory log (every stage change, timestamped).
  • Core endpoints: list all jobs · get one job with candidates · get one application with full stage history · update a candidate's stage · list all rejected candidates.
  • Stage transition validation — the API must reject nonsensical moves (e.g. Accepted → Applied). Document your allowed transition map in the README.
Frontend
  • React (preferred) or Vue. No UI library required — how you write CSS is part of the evaluation.
  • All four screens must be functional and connected to the real API. No hardcoded mock data at submission.
  • Drag-and-drop on the Kanban board must call the backend. Stage changes must persist — not just update local state.
  • Deployment — Host your frontend on Vercel, Netlify, or similar. The API must be accessible (Render, Railway, Fly.io, etc.). Include live URLs in README.
Both
  • Loading and error states — the UI must show when data is loading and handle API errors gracefully. Silent failures are not acceptable.
  • Date & time tracking — every stage change records a precise timestamp. Application date and rejection date/time are always visible where relevant.
  • Environment configuration — Your README must clearly document all env vars. We will test your deployment.
Bonus
  • Seed script with realistic dummy data (3+ jobs, 10+ candidates) — makes the demo far more compelling.
  • Optimistic UI on drag — the card moves immediately and rolls back if the API returns an error.
  • Any unit test, however small — validation logic, a stage transition check, anything that shows you thought about correctness.
